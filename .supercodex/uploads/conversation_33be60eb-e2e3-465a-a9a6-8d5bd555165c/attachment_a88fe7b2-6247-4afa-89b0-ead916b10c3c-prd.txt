## 1. 产品愿景

> **让多 Agent 协作像一家"看得见的公司"在运作。**

AgentVille 是一个**可视化的多 Agent 协作平台**。用户在这里组建 AI 团队、派发任务、观察协作过程；外部 AI 工具（Claude Code、Codex 等）可以把 AgentVille 的团队当作"外包公司"直接调用。

我们相信下一阶段的 AI 生产力来自**组织化的 Agent 协作**，但当前的多 Agent 系统普遍存在三个问题：黑盒、难调试、无趣。AgentVille 用"AI 公司"这一隐喻把它们一次性解决。

---

## 2. 产品定位

### 2.1 一句话定位
**一个把多 Agent 协作可视化为 AI 公司的协作平台，并以 MCP 标准对外暴露团队能力。**

### 2.2 三个定位锚点

| 锚点 | 含义 | 与同类产品的差异 |
|---|---|---|
| **组织化** | Agent → Team → Multi-Team 三层结构，有角色、有 Leader、有协作规则 | 不只是"Agent 列表"，而是"组织架构" |
| **可视化** | 办公室俯视视图 + 节点图调试视图双视角 | 不是抽象拓扑图，是有"人"在动的场景 |
| **可外包** | 通过 MCP 把整支团队暴露给外部 Agent 调用 | 不只是用工具，更是"被工具使用" |

### 2.3 不是什么

- **不是又一个 AutoGen / CrewAI** —— 那些是框架（给开发者写代码），AgentVille 是产品（给用户用界面）
- **不是 Agent Marketplace** —— 不卖 Agent，专注协作与编排
- **不是通用聊天机器人** —— 任务驱动，不是闲聊驱动

## 3. 目标用户与场景

### 3.1 用户画像

| 画像 | 描述 | 核心诉求 | 价值主张 |
|---|---|---|---|
| **AI 重度使用者（P0）** | 已经在用 Claude Code/Codex 等工具开发的工程师 | 想把重复的"调研+整理+输出"流程外包，专注核心代码 | 让 Claude Code 直接调用 AgentVille 团队，把杂活外包 |
| **效率型知识工作者（P1）** | 产品经理、运营、咨询、研究员 | 想用多 Agent 完成"调研→分析→成文"链路，但缺乏编程能力 | 可视化组队 + 自然语言派活，不写代码 |
| **AI 应用开发者（P1）** | 在做 Agent 类产品的同行 | 需要一个可复用的多 Agent 编排底座 | 开放 API/SDK，把 AgentVille 嵌入自己的产品 |
| **AI 爱好者 / 玩家（P2）** | 喜欢调教 AI、看 AI 互动的用户 | 想看 AI 协作的过程，享受"养成"乐趣 | 办公室视图、AI 员工人格化、招聘/解雇 |

### 3.2 核心场景（按优先级）

#### S1：外部 Agent 外包任务（P0，差异化核心）
> 工程师在 Claude Code 写代码时，让代码助手把"帮我调研一下竞品的 X 功能"丢给 AgentVille 的"调研团队"，自己继续写代码。完成后查看产出。

#### S2：浏览器内组队执行任务（P0）
> 用户登录 AgentVille，组建一个"研报团队"（含分析师、数据工程师、文字编辑），自然语言下达任务"产出一份 Q2 行业研报"，实时观察 Agent 协作过程，最终获得文档。

#### S3：观察与调试（P0）
> 开发者在节点图视图中查看上一次任务执行链路，定位某个 Agent 为何做出错误决策，调整其 prompt 或工具权限后重跑。

#### S4：多团队协作（P1）
> "研报团队"的 Leader 在执行中向"数据团队"请求一份数据，两支团队的 Leader 自动握手协作。

#### S5：观赏与养成（P2）
> 用户在闲暇时打开办公室视图，看自己的 AI 员工在跑任务，给优秀员工"加薪"（更好的模型）或"换岗"。

## 4. 产品核心概念

| 概念 | 定义 | 用户层认知 |
|---|---|---|
| **Agent（员工）** | 单个 AI 实例，有角色、能力、人格、可用工具 | "我的 AI 员工" |
| **Team（团队/部门）** | 一组 Agent + 一个 Leader + 协作规则 | "我的 AI 团队" |
| **Leader（团队领导）** | 团队中负责规划、派活、裁决的特殊 Agent | "团队负责人" |
| **Office（办公室）** | 一个 Team 的可视化空间 | "我的办公室" |
| **Task（任务）** | 用户或外部系统派发给 Team / Agent 的工作 | "派活" |
| **Run（执行）** | Task 的一次具体执行实例 | "这次跑" |
| **Capability（能力）** | Agent 可以做的事的原子声明 | "技能" |
| **Tool（工具）** | Agent 可以调用的外部能力（搜索、画图等） | "工具" |
| **Memory（记忆）** | 短期对话、团队共享笔记、长期经验 | "记忆 / 团队共享文档" |
| **Persona（形象）** | Agent 的外观与人格设定 | "员工卡片" |

## 6. 关键用户旅程

### 6.1 首次上手（Onboarding）

```
1. 注册登录
   ↓
2. 引导：选一个团队模板（"研报团队" / "代码 Review 团队" / "调研团队"）
   ↓
3. 自动生成 3 个 AI 员工，进入办公室视图
   ↓
4. 引导派一个示例任务（"调研一下 2026 年 AI Agent 行业现状"）
   ↓
5. 实时观察员工协作 5 分钟内出结果
   ↓
6. 引导：连接 Claude Code（MCP 配置一键复制）
```

**北极星目标**：用户在 **10 分钟内** 完成"创建团队→派活→拿到产物→连上 Claude Code"的完整闭环。

### 6.2 日常使用：浏览器内派活

```
用户进入 Workspace
   ↓
选择/切换团队 → 在派活框输入自然语言任务
   ↓
（可选）选择交付格式、设置预算上限、指定截止时间
   ↓
点击"派活" → 跳转到任务详情页（默认 Office View）
   ↓
观察 Agent 协作（可切换 Graph View 看链路）
   ↓
任务完成 → 通知 + 产物列表
   ↓
查看 / 下载 / 二次编辑 / 重跑
```

### 6.3 关键场景：外部 Agent 外包

```
用户在 Claude Code 中输入：
"@agentville 调研一下 Rust 异步 actor 框架的现状，对比 Actix / Coerce / Riker"
   ↓
Claude Code 调用 av.team.submit (team="调研团队", task="...")
   ↓
AgentVille 立即返回 task_id，并在用户的 Workspace 通知中心出现新任务
   ↓
用户可选：
  (a) 继续在 Claude Code 中等待（Claude Code 后台轮询 av.task.status）
  (b) 跳转到 AgentVille Web 实时观察
   ↓
任务完成，Claude Code 收到产物 URL + 摘要
   ↓
用户在 Claude Code 中继续基于产物写代码
```

## 7. 核心功能详述

### 7.1 团队管理

**团队创建**
- 三种入口：从模板创建 / 从零创建 / 复制现有团队
- 必填：团队名、目标领域（决定默认工具集）
- 可选：成员数量上限、预算上限、是否允许跨团队通信

**员工招聘**
- 选 Role（Engineer / Analyst / Editor / Researcher / Custom）
- 选 Persona（头像、名字、性格描述、口头禅）
- 配置 LLM Model（自动按角色推荐，可手动覆盖）
- 工具授权（按 Capability 勾选，超出预设需二次确认）
- 设置个性化 prompt（高级）

**Leader 配置**
- 必须指定一名（默认创建团队时自动生成）
- 选择策略：Sequential / Reactive / Hybrid（人话版："按计划走" / "随机应变" / "混合"）
- 设置 Leader 是否有"招聘权"（自动扩员）

**工具授权**
- 三类来源：内置工具、MCP 远程工具、外部 Agent
- 默认按 Role 给一套白名单
- 危险工具（写文件、执行 shell、调用付费 API）必须显式开启

### 7.2 任务派发与执行

**派活入口**
- 自然语言输入框（主入口）
- 结构化派活（高级：指定具体步骤、限定工具、设置子任务依赖）
- API/MCP 派活（外部系统）
- 定时派活（V2）

**派活时可选项**
- 目标 Team / 单 Agent
- 预算上限（token / $）
- 截止时间
- 交付格式（文档 / 表格 / 代码 / 图片 / 自由）
- 重要性（影响模型选择策略）
- 上下文附件（可上传文件作为参考）

**任务生命周期可视化**
- 状态条：Pending → Planning → Running → Done
- 每步实时更新：当前哪个 Agent 在做什么、用了什么工具、花了多少 token
- 支持随时**暂停 / 继续 / 取消 / 注入指令**（"停一下，先把 X 做掉"）

**产物管理**
- 自动按类型聚合：文档、表格、图片、代码、数据
- 一键导出（Markdown / PDF / Word / 飞书文档）
- 与历史版本对比
- 二次派活（"在此基础上再补一段竞品分析"）

### 7.3 双视图可视化

#### Office View（办公室俯视，主视图）
- 像素风 / 卡通风 2D 俯视场景
- 每个 Agent 是一个员工形象，有座位、有动画
- 状态映射：
  - 空闲 → 在工位上喝咖啡 / 看书
  - 思考中 → 头顶冒灯泡
  - 调工具 → 走到对应工具区（如"打印机"=文件输出）
  - 与同事沟通 → 走到对方工位，弹出对话气泡
  - 与 Leader 汇报 → 走到 Leader 办公室
- 点击 Agent → 右侧抽屉查看详细状态、当前 prompt、最近 tool calls
- 点击 Leader → 查看当前 Plan 和分工

#### Graph View（节点图，调试视图）
- 默认按 Run 展开：节点 = Agent / Step，边 = 消息
- 节点上显示：状态、耗时、token、$
- 点击节点 → 查看完整 prompt / response / tool call 详情
- 支持过滤、搜索、按时间倒放

#### Timeline View（时间线）
- 横轴时间，纵轴 Agent
- 每个 Agent 的活动以色块呈现（思考 / 调工具 / 等待 / 通信）
- 便于发现瓶颈 Agent 或长 idle 段

#### Cost View（成本视图）
- 按 Team / Agent / Model / 时间段聚合
- 显示 token 消耗、$ 消耗、调用次数
- 预算预警与超额拦截

### 7.4 集成中心

#### MCP Server（核心差异化）
- 启用后，AgentVille 自动暴露标准 MCP endpoint
- 提供一键复制的 Claude Code / Codex 配置片段
- 可以选择性暴露：哪些 Team / Agent 对外可见
- 调用时强制鉴权（API Key 或 OAuth）
- 调用配额与计费可控

#### MCP Client
- 配置外部 MCP servers（如官方 GitHub MCP、Slack MCP）
- 配置后，工具自动出现在 Agent 的可授权列表
- 支持 server 健康检查与版本管理

#### External Agent 入职
- 在"招聘"流程中选择"外部 Agent"
- 配置 Claude Code / Codex 的启动参数与工作目录
- 设置资源限制（CPU、内存、超时）
- 入职后该外部 Agent 在 Office View 中以特殊形象出现（"外包员工"）

#### A2A 互通
- 启用后，每个 Team Leader 暴露 A2A endpoint
- 可与其他 AgentVille 实例或第三方 A2A 系统互通
- 强制走 Inter-team Gateway（配额、审计、环路检测）

### 7.5 治理

#### 预算与配额
- 三层预算：组织 / 团队 / 单任务
- 软预算（warn）+ 硬预算（block）
- 超额时的策略：拦截 / 降级模型 / 通知人类

#### RBAC
- 角色：Owner / Admin / Member / Viewer
- 资源粒度：组织 / 团队 / 任务 / 集成
- 工具维度：危险工具需要 Admin 显式启用

#### 审计日志
- 所有派活、Agent 决策、工具调用、跨 Team 通信都可追溯
- 提供按用户 / 团队 / 时间过滤的查询
- 可导出（合规需求）

## 8. UI / UX 设计原则

1. **双视图并存，默认 Office，一键切 Graph** —— 用户层"有趣"，开发者层"可调"
2. **AI 员工有人格** —— 名字、头像、性格、对话气泡有情绪，但不能喧宾夺主
3. **任务永远透明** —— 任意时刻用户都能知道"现在哪个 Agent 在做什么、为什么"
4. **进度感清晰** —— 长任务必须显示进度阶段、预估剩余时间、当前花费
5. **干预要简单** —— 暂停、继续、注入指令、取消，三步以内可达
6. **错误要友好** —— Agent 失败要给出"为什么 + 建议"，而不是堆栈
7. **空状态有引导** —— 没团队？引导建团队。没任务？引导派活。
8. **键盘友好** —— 派活、切换 Team、查任务都有快捷键（重度用户高频）
9. **可观测优先** —— 即便牺牲一点"萌"也要保证可调试
10. **国际化与隐私优先** —— 默认不外发数据，可选自建 LLM 后端

## 9. 衡量指标

### 9.1 北极星指标
**Weekly Active Tasks（WAT）**：周内被实际派发并完成的任务数。直接反映"产品是否在解决真实问题"。

### 9.2 核心指标

| 维度 | 指标 | 目标（V1 上线 3 个月） |
|---|---|---|
| **获客** | 周新增用户 | — |
| **激活** | 首次任务完成率（从注册到第一个 Task Succeeded） | ≥ 60% |
| **激活** | 首任务 TTV（Time to Value，从注册到第一个产物） | ≤ 10 min |
| **留存** | 次周留存 / 4 周留存 | ≥ 40% / ≥ 20% |
| **使用深度** | 人均周派活数 | ≥ 5 |
| **集成** | MCP 接入率（注册用户中接入 Claude Code/Codex 的比例） | ≥ 30% |
| **集成** | 外部派活占比（来自 MCP 的任务 / 总任务） | ≥ 25% |
| **质量** | 任务成功率 | ≥ 80% |
| **质量** | 用户主动取消率 | ≤ 10% |
| **成本** | 单任务平均 token | 持续下降趋势 |
| **NPS** | 月度 NPS | ≥ 30 |

### 9.3 反指标（不能为了上面指标牺牲的）
- 任务时长（不能为了成功率而无限重试）
- 用户主动干预率（不能让用户疲于救火）
- 产物质量评分（任务"完成"不等于"做好了"）

## 10. 商业模式（早期假设）

| 套餐 | 定价 | 适合 | 包含 |
|---|---|---|---|
| Free | $0 | 体验 | 1 团队 / 3 员工 / 50 任务每月 / 自带 API key |
| Pro | $20/月 | 个人深度用户 | 5 团队 / 20 员工 / 1000 任务 / 内置 token 包 / MCP Server |
| Team | $80/座位/月 | 小团队 | 不限团队 / 不限员工 / SSO / 审计日志 / A2A |
| Enterprise | 定制 | 大型组织 | 私有部署 / VPC / 自建模型接入 / SLA |

**关键变量**：LLM token 成本。早期建议"自带 key" 模式降低风险，逐步演进到"含 token 包"。

## 11. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| 多 Agent token 成本指数级膨胀 | 用户用不起 | Cost View + 模型分级 + 预算硬上限 + 摘要压缩 |
| 任务质量不稳定（同一任务两次结果差异大） | 信任崩塌 | 模板化 Team + 提示词版本管理 + 回放与对比 |
| 可视化喧宾夺主，看着热闹用着废 | 留存差 | 双视图严格分工 + 默认任务详情页优先信息密度 |
| MCP / A2A 协议演进快，集成易碎 | 兼容性问题 | 协议层独立抽象，对外保持稳定 contract |
| Rust 开发慢，迭代不过竞品 | 错过窗口 | Prompt 外置热更新 + Mock 测试 + 双周 release 节奏 |
| 外部 Agent（Claude Code/Codex）协议变更 | 集成失效 | Adapter 层隔离 + 定期回归测试 + 多 Adapter 并存 |
| 安全：Agent 误调危险工具 | 数据/资金损失 | 危险工具白名单 + 二次确认 + 沙箱执行 + 审计 |
| 用户期望被科幻片拉高，实际能力跟不上 | 流失 | Onboarding 设定正确预期 + 模板用真实可完成的任务 |
